# 数据库索引优化方案

## 1. 问题诊断

### 当前慢查询原因

| 原因 | 影响 |
|------|------|
| Redis 未启动，每次请求卡 5 秒超时再 fallback MySQL | 响应延迟 +5s |
| `transaction_history` 60 万行，GROUP BY / COUNT DISTINCT 全表扫描 | 单次查询 3-8s |
| `device_id`、`counterparty_id` 无索引，聚合查询全表扫描 | 设备/收款方页面 5s+ |
| `user_profile` 筛选（risk_score / account_status）无索引 | 高风险用户页 1-2s |
| `risk_alert` 按 create_time 分组统计无覆盖索引 | Dashboard 每次刷新 1-2s |

## 3. 推荐索引

```sql
-- ============================================================
-- transaction_history（60 万行，查询最频繁）
-- ============================================================

-- 1. 时间范围查询（覆盖最多查询场景）
--    所有 "WHERE trans_timestamp BETWEEN" / "WHERE trans_timestamp > ?"
--    以及 GROUP BY DATE(trans_timestamp), GROUP BY city WHERE trans_timestamp > ?
ALTER TABLE transaction_history ADD INDEX idx_trans_ts (trans_timestamp);

-- 2. 设备聚合查询
--    GROUP BY device_id + COUNT DISTINCT user_id + COUNT(*)
--    COUNT DISTINCT device_id
ALTER TABLE transaction_history ADD INDEX idx_device (device_id);

-- 3. 收款方聚合查询
--    GROUP BY counterparty_id + SUM(amount) + COUNT DISTINCT user_id
--    COUNT DISTINCT counterparty_id
ALTER TABLE transaction_history ADD INDEX idx_cpty (counterparty_id);

-- 4. 用户维度查询
--    COUNT DISTINCT user_id WHERE trans_timestamp BETWEEN
ALTER TABLE transaction_history ADD INDEX idx_user_ts (user_id, trans_timestamp);

-- 5. 渠道/类型 GROUP BY（小表，可选）
ALTER TABLE transaction_history ADD INDEX idx_pay_channel (pay_channel);
ALTER TABLE transaction_history ADD INDEX idx_trans_type (trans_type);

-- ============================================================
-- user_profile（4.9 万行）
-- ============================================================

-- 1. 高风险用户筛选：WHERE risk_score >= 80
ALTER TABLE user_profile ADD INDEX idx_risk_score (risk_score);

-- 2. 账户状态筛选
ALTER TABLE user_profile ADD INDEX idx_account_status (account_status);

-- ============================================================
-- risk_alert（增长中，实时查询）
-- ============================================================

-- 1. 时间范围统计（Dashboard / 趋势图 / 城市分布）
--    WHERE create_time > ? GROUP BY risk_level
--    WHERE create_time > ? GROUP BY hit_rules
ALTER TABLE risk_alert ADD INDEX idx_create_time (create_time);

-- 2. 风险等级筛选
ALTER TABLE risk_alert ADD INDEX idx_risk_level (risk_level);

-- 3. 告警城市分布
--    WHERE create_time > ? AND risk_level IN (...) GROUP BY alert_loc
ALTER TABLE risk_alert ADD INDEX idx_alert_loc (alert_loc);

-- ============================================================
-- t_metrics
-- ============================================================

-- 趋势图时间范围查询
ALTER TABLE t_metrics ADD INDEX idx_snapshot_time (snapshot_time);
```

## 4. 索引效果预估

| 查询 | 加索引前 | 加索引后 |
|------|:---:|:---:|
| `COUNT WHERE trans_timestamp BETWEEN` | 全表扫描 60万行 ~3s | 索引扫描 ~0.05s |
| `GROUP BY device_id` | 全表扫描 + 临时表排序 ~5s | 索引扫描 ~0.3s |
| `GROUP BY counterparty_id` | 全表扫描 + 临时表排序 ~5s | 索引扫描 ~0.3s |
| `COUNT DISTINCT device_id` | 全表扫描 ~2s | 索引扫描 ~0.02s |
| `risk_score >= 80` (user_profile) | 全表扫描 4.9万行 ~0.3s | 索引扫描 ~0.01s |
| `GROUP BY risk_level WHERE create_time > ?` | 全表扫描 ~1s | 索引扫描 ~0.05s |

**综合效果：离线分析页面从 5-10 秒降到 0.5 秒以内。**

## 6. 注意事项

1. **加索引期间表会锁住**。建议在低峰期执行，或使用 `ALGORITHM=INPLACE, LOCK=NONE`（MySQL 5.6+ 支持）
2. **索引占用磁盘空间**。6 个索引预计增加 50-100MB
3. **写入性能会略微下降**。INSERT 时需要更新索引，但风控场景读多写少，影响可忽略
4. **先解决 Redis 问题**。Redis 启动后响应已经能快一大截，索引是在此基础上的进一步优化
