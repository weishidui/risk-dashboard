# 金融交易风险实时监控与离线分析平台

## 需求分析文档

---

## 1. 项目概述

本项目面向模拟金融转账场景，已经实现以下闭环：

1. 生成历史交易与实时交易；实时交易同时进入 MySQL 和 Kafka。
2. Spark Streaming 消费 Kafka `trans-event`，为每笔交易执行规则评分，维护资金链路、实时指标和告警。
3. MapReduce 从 MySQL 同步历史数据并进行 ODS、DWD、DWS、ADS 分层计算，产出用户画像和离线风险分析结果。
4. Spring Boot + Vue 大屏分别展示实时监控和离线分析，并支持告警处置、用户画像、设备/收款方/资金链路查询。

```text
历史数据生成器 ──> MySQL transaction_history 等源表
                                  │
```

## 2. 数据接入需求

### 2.1 历史数据

- `history_bulk_loader.py` 生成历史交易并写入 MySQL；该脚本不发送 Kafka。
- 历史原始交易保存于 `transaction_history`，用于后续离线分析。
- 离线全流程脚本按分析日期 `dt` 构造时间窗口：`[dt - 30 天 00:00:00, dt 当天 00:00:00)`，即分析日前 30 天且不包含 `dt` 当天。

### 2.2 实时交易

- 网关 `risk-gateway` 接收交易后，写入 MySQL `transaction_history`，并发送 JSON 报文至 Kafka Topic `trans-event`。
- Kafka 消息以 `user_id` 为消息 key；实时引擎消费者组为 `risk-streaming-engine`，启动参数为 `--kafka-offset-reset latest`，消费该消费者组启动后的新消息。
- Kafka-to-HDFS 程序可独立消费同一 Topic，落盘路径为：

```text
/user/hive/warehouse/risk_control.db/ods_trans_event/dt=yyyyMMdd/source=kafka/part-kafka-*.jsonl
```

- MySQL 离线导入路径与实时 Kafka 落盘通过 `source` 目录区分：

```text
/user/hive/warehouse/risk_control.db/ods_trans_event/dt=yyyyMMdd/source=sqoop
```

### 2.3 原始交易字段

`transaction_history` 物理表有 33 列：`id`、`create_time` 两个系统列，加 31 个业务列。Kafka 交易报文使用 `timestamp`，入 MySQL 时对应 `trans_timestamp`。

| 分组 | 实际字段 |
|---|---|
| 标识与金额 | `trans_id`、`user_id`、`amount`、`trans_timestamp` |
| 位置与设备 | `city`、`geo_location`、`device_id`、`network_type`、`dev_score`、`ip_address`、`os_type`、`os_version`、`screen_resolution`、`battery_level`、`root_jailbreak`、`sim_operator`、`user_agent`、`dns_server`、`wifi_ssid` |
| 交易行为 | `trans_type`、`pay_channel`、`input_method`、`click_duration`、`note`、`page_url` |
| 收款方 | `counterparty_id`、`counterparty_name`、`counterparty_bank` |
| 会话 | `login_session_id`、`login_fail_count`、`cancel_retry_count` |

## 3. 离线分析需求

### 3.1 分层与数据格式

离线作业使用 Hadoop MapReduce。ODS、DWD、DWS、ADS 文件均采用一行一个 JSON 的 `TextOutputFormat`。

| 层级 | 路径或表 | 已实现处理 |
|---|---|---|
| ODS | `ods_trans_event`、`ods_login_event`、`ods_trader_user`、`ods_trader_counterparty`、`ods_counterparty_blacklist`、`ods_trans_chain`、`ods_history_risk_seed_log` | 从 MySQL 读取原始 JSON；交易与维表按 `dt` 分区 |
| DWD | `dwd_trans_event` | 清洗、字段标准化、空值处理、按 `trans_id` 去重 |
| DWS | `dws_user_transaction_daily`、`dws_counterparty_daily`、`dws_device_daily`、`dws_user_risk_daily` | 用户、收款方、设备、历史风险种子聚合 |
| ADS | `ads_user_profile`、`ads_transaction_risk_detail`、`ads_risk_dashboard`、`ads_cross_region_risk_flow` | 用户画像、交易风险明细、离线看板指标、跨区域风险流 |

### 3.2 离线任务与产物

| 作业 | 输入 | 输出/副作用 |
|---|---|---|
| `mysql-to-ods` | `transaction_history` | 交易 ODS JSON |
| `mysql-table-to-ods` | 指定业务表 | 对应维表 ODS JSON |
| `ods-to-dwd` | 交易 ODS | 清洗交易 DWD |
| `dwd-to-dws-user` | DWD | 用户交易日汇总 |
| `dwd-to-dws-cpty` | DWD | 收款方日汇总，同时更新 `counterparty_blacklist` 和 Redis `counterparty:{counterparty_id}` |
| `dwd-to-dws-device` | DWD | 设备日汇总，同时更新 Redis `device_shared:{device_id}`、`device_risk:{device_id}` |
| `history-risk-to-dws-user-risk` | 交易 ODS + 历史风险种子 ODS | 用户风险日汇总 |
| `dws-to-user-profile` | 用户 DWS、账户、登录、用户风险 DWS | HDFS 用户画像 ADS |
| `ads-user-profile-to-mysql` | 用户画像 ADS | upsert `user_profile` |
| `ads-transaction-risk-detail` | DWD + 历史风险种子 | 交易风险明细 ADS |
| `ads-risk-dashboard` | 风险明细 ADS | 看板指标、评分分布、地区/城市/规则/时间/行为/特征排名 ADS |
| `ads-cross-region-flow` | 风险明细 ADS | 跨省市风险流 ADS |
| `ads-risk-to-mysql` | ADS HDFS 结果 | 写入 14 张 `ads_*` MySQL 表 |

### 3.3 离线分析结果

离线看板读取 MySQL ADS 快照表，不在页面请求时扫描原始交易表。分析结果包括：

- 概览：交易总数、风险交易数、高危数、极度危险数、用户/设备/收款方去重数、金额和平均风险评分。
- 评分桶分布。
- 省份、城市、用户、设备、收款方风险排行。
- 规则命中排行与规则类别。
- 按小时风险趋势。
- 风险行为分布：交易渠道、交易类型、输入方式等由 ADS 结果提供。
- 风险特征分布：总量、风险量、高危量、极度危险量、风险率、高危率、风险金额、平均分。
- 高风险交易明细和跨区域风险流。

## 4. 实时风控需求

### 4.1 实时处理逻辑

实时风控引擎在 Spark 集群执行 `RiskStreamingApp`，每 100 ms 微批消费 Kafka `trans-event`。每笔事件的处理顺序为：

1. 优先从 Redis `profile:{user_id}` 与 `counterparty:{counterparty_id}` 读取画像和收款方风险；缓存缺失时回退查询 MySQL。
2. 更新 Redis 实时状态：同 IP 登录失败用户集合、同设备用户集合、60 秒用户交易计数、小额测试计数。
3. 查询并写入 `trans_chain`，计算回流、链路深度、30 分钟汇聚/分散计数。
4. 执行 A-I 九类共 60 条规则，累计分值并计算风险等级和处理决策。
5. 写入 Redis 10 秒实时指标桶；对命中至少一条规则的交易通过 HTTP 写入 `risk_alert`，由后端向 WebSocket 推送。

### 4.2 风险等级与处理决策

| 项目 | 实际实现 |
|---|---|
| 低危 | `finalScore < 41` 且不属于硬阻断；也可能是命中低分规则后的等级 |
| 中危 | `41 <= finalScore < 71` |
| 高危 | `71 <= finalScore <= 120` |
| 极度危险 | `finalScore > 120`，或命中 F1 高危收款方黑名单硬阻断 |
| 放行 | 无硬阻断且 `finalScore < 41` |
| 人工复核 | `41 <= finalScore <= 120` |
| 拦截 | 硬阻断或 `finalScore > 120` |

`risk_alert` 的写入前提是命中至少一条规则；评分为 0、完全未命中规则的交易不进入 `risk_alert`。

### 4.4 Redis 数据需求

| Key 模式 | 写入方 | 实际用途/生命周期 |
|---|---|---|
| `profile:{user_id}` | 离线画像作业 | 实时规则读取用户 30 天均额、常用属性、限额、登录和风险标签；无显式 TTL 时持久 |
| `counterparty:{counterparty_id}` | 离线收款方作业 | 实时读取收款方等级、标签、收款金额、付款人数、注册天数 |
| `device_shared:{device_id}` | 离线设备作业 | 标识共享设备，TTL 由离线配置控制 |
| `device_risk:{device_id}` | 离线设备作业 | 存储设备风险汇总 |
| `rt:*` | 实时引擎 | IP、设备、用户频次与小额测试状态，按代码设置 60 秒至 30 天不等 TTL |
| `risk:rt:metric:10s:{bucket}` | 实时引擎 | 10 秒指标 Hash，TTL 48 小时 |
| `risk:rt:users:10s:{bucket}` | 实时引擎 | 10 秒活跃用户 HyperLogLog，TTL 48 小时 |
| `risk:rt:city:10s:{bucket}` | 实时引擎 | 10 秒城市计数 Hash，TTL 48 小时 |
| `risk:rt:rule:10s:{bucket}` | 实时引擎 | 10 秒规则命中 Hash，TTL 48 小时 |
| `risk:rt:dedup:{trans_id}` | 实时引擎 | 指标去重，TTL 48 小时 |

实时看板读取连续 6 个 10 秒桶，即最近 60 秒。风险指数公式为：

```text
min(100, 平均风险分*0.45 + 高危及极度危险占比*0.30 + 拦截率*0.25)
```

每分钟，后端将上一个完整分钟的实时聚合写入 MySQL `t_metrics`，用于历史趋势。

## 6. 数据库结构

系统使用 `risk_control` 数据库保存业务数据、实时结果和离线分析结果。

### 6.1 业务源表与实时表

| 表 | 主键/唯一约束 | 字段 |
|---|---|---|
| `transaction_history` | `id` 主键，`trans_id` 唯一 | `id BIGINT`、`trans_id VARCHAR(64)`、`user_id VARCHAR(64)`、`amount DOUBLE`、`trans_timestamp BIGINT`、`city VARCHAR(64)`、`geo_location VARCHAR(64)`、`device_id VARCHAR(128)`、`network_type VARCHAR(32)`、`dev_score INT`、`create_time TIMESTAMP`、`ip_address VARCHAR(45)`、`os_type VARCHAR(20)`、`os_version VARCHAR(30)`、`screen_resolution VARCHAR(20)`、`battery_level INT`、`root_jailbreak TINYINT(1)`、`sim_operator VARCHAR(20)`、`user_agent VARCHAR(500)`、`dns_server VARCHAR(45)`、`wifi_ssid VARCHAR(100)`、`trans_type VARCHAR(20)`、`pay_channel VARCHAR(20)`、`input_method VARCHAR(20)`、`click_duration BIGINT`、`note VARCHAR(200)`、`page_url VARCHAR(200)`、`counterparty_id VARCHAR(64)`、`counterparty_name VARCHAR(64)`、`counterparty_bank VARCHAR(64)`、`login_session_id VARCHAR(64)`、`login_fail_count INT`、`cancel_retry_count INT` |
| `trader_user` | `id` 主键，`user_id`、`username` 唯一 | `id`、`user_id`、`username`、`password_hash`、`real_name`、`phone`、`account_status`、`total_balance`、`single_limit`、`daily_limit`、`monthly_limit`、`default_city`、`default_device_id`、`default_pay_channel`、`registration_time`、`create_time`、`update_time`、`risk_tags` |
| `login_event` | `id` 主键 | `id`、`user_id`、`login_time`、`ip_address`、`login_fail`、`session_id`、`create_time` |
| `trader_counterparty` | `id` 主键 | `id`、`user_id`、`counterparty_id`、`counterparty_name`、`counterparty_bank`、`is_frequent`、`create_time` |
| `counterparty_blacklist` | `id` 主键，`counterparty_id` 唯一 | `id`、`counterparty_id`、`counterparty_name`、`risk_level`、`risk_type`、`source`、`total_received_24h`、`total_received_7d`、`unique_payers_24h`、`registration_days`、`risk_tags`、`created_time`、`update_time`、`fast_in_out_score`、`status` |
| `history_risk_seed_log` | `id` 主键 | `id`、`trans_id`、`risk_code`、`risk_name`、`risk_category`、`risk_score`、`seed_note`、`create_time` |
| `trans_chain` | `id` 主键 | `id`、`chain_id`、`trans_id`、`user_id`、`counterparty_id`、`prev_trans_id`、`hop_order`、`chain_depth`、`is_loop`、`chain_type`、`amount`、`create_time` |
| `user_profile` | `id` 主键，`user_id` 唯一 | `id`、`user_id`、`avg_amt_30d`、`common_cities`、`common_devs`、`last_trans_ts`、`last_city`、`update_time`、`common_pay_channels`、`common_trans_types`、`common_counterparties`、`last_ip`、`last_login_time`、`registration_time`、`total_balance`、`single_limit`、`daily_limit`、`monthly_limit`、`account_status`、`login_count_24h`、`trans_count_24h`、`trans_amount_24h`、`trans_count_7d`、`cancel_retry_count`、`risk_tags`、`risk_score` |
| `risk_alert` | `id` 主键 | `id`、`alert_id`、`trans_id`、`user_id`、`amount`、`city`、`hit_rules`、`final_score`、`risk_level`、`alert_loc`、`raw_json`、`create_time`、`counterparty_id`、`ip_address`、`is_new_device`、`is_new_counterparty`、`chain_id`、`status`、`handler`、`handle_time`、`handle_remark` |
| `t_metrics` | `id` 主键 | `id`、`snapshot_time`、`total_transactions`、`pass_count`、`verify_count`、`block_count`、`high_risk_count`、`medium_risk_count`、`low_risk_count`、`avg_risk_score`、`env_risk_count`、`amount_risk_count`、`teleport_risk_count`、`geo_risk_count`、`avg_latency` |
| `sys_user` | `id` 主键，`username` 唯一 | `id`、`username`、`password`、`role`、`create_time` |
